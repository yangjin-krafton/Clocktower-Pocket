# Clocktower Pocket

오프라인 소셜 디덕션 보드게임 **“시계탑에 흐른 피(Blood on the Clocktower)”** 를 사내에서 진행할 때, 종이/토큰/그리모어 같은 **물리 소품을 휴대폰 웹 UI로 대체**하기 위한 **현장 보조용 웹 앱 샘플**입니다.
게임의 핵심(현장 토론, 처형, 밤/낮 흐름, 스토리텔러 진행)은 그대로 두고, **정보 전달/투표/타이머/기록**만 디지털로 보조합니다. ([Blood on the Clocktower Wiki][1])

> ⚠️ 비공식 팬메이드/사내 PoC입니다. 원작 게임과 혼동되지 않도록 **로고/아트/역할 텍스트** 등 저작권 소지가 있는 리소스는 포함하지 않습니다.
> 역할 능력 텍스트는 요약/패러프레이즈 또는 내부 링크(공식 위키/참조) 형태로 처리합니다. ([Blood on the Clocktower Wiki][2])

---

## 목표(Goals)

* 참가자는 **개인 휴대폰 브라우저**로 접속해 자신의 상태(생존/사망), 알림, 투표, (허용되는 범위의) 개인 정보 등을 확인
* 스토리텔러는 **관리자 화면(그리모어 대체)** 에서 배역/밤 액션/사망/처형/투표 결과를 관리
* 네트워크는 사내 행사 특성을 고려해 **로컬(동일 Wi-Fi) 단일 서버**로 운영

## 비목표(Non-goals)

* 온라인(원격) 완전 자동 진행 게임을 목표로 하지 않음(토론/심리전은 오프라인 현장 중심)
* “정식 룰북/정식 에셋” 제공 및 배포 목적이 아님

---

## 게임 진행 개요(현장 흐름)

본 샘플은 기본적으로 아래 흐름을 지원합니다.

* **밤:** 스토리텔러가 순서대로 역할 능력을 처리(플레이어는 휴대폰으로 “호출/선택/확인”만 수행 가능)
* **낮:** 현장 토론 → 지명/투표 → 처형 처리(웹에서 투표 수집/집계/로그)
* 일반적으로 **선 팀은 데몬 처형**, **악 팀은 종료 조건(예: 생존 3인)** 달성에 목표 ([Blood on the Clocktower Wiki][1])

> 입문 스크립트는 Trouble Brewing을 기본 타겟으로 가정(필요 시 스크립트 교체) ([Blood on the Clocktower Wiki][3])

## 시스템 구성(Architecture)

### 1) 클라이언트(참가자)

* 모바일 우선(Responsive) 웹 UI
* 기능

  * “나의 상태”: 생존/사망, 남은 투표권(사망 후 유령 투표 1회 등 룰 옵션)
  * “알림”: 밤 행동 요청/확인, 낮 투표 시작/종료, 처형 결과
  * “투표”: 찬/반 또는 지명자 선택(진행 방식에 따라)
  * “비공개 정보”: 본인 역할(또는 스토리텔러 설정에 따른 비공개 정보 표시)

### 2) 관리자(스토리텔러)

* 데스크톱/태블릿 최적
* 기능

  * 방 생성/코드 발급, 플레이어 등록/좌석(원형 시계 방향) 관리
  * 배역 배정(랜덤/수동), “블러프(악팀 가짜 역할)” 메모
  * 밤 페이즈 진행(액션 큐/프롬프트)
  * 사망/처형 처리, 투표 집계/결과 확정
  * 로그(밤 행동/정보 제공 내역), 되돌리기(Undo) 옵션

### 3) 서버

* 사내 행사 기준: **Node.js + Express**(예시) 또는 사내 선호 스택
* 실시간: **WebSocket(Socke.IO 등)** 또는 SSE
* 데이터: 인메모리(샘플) → SQLite/Redis(확장)
* 접근: 로컬 IP + 포트(예: `http://10.0.0.12:8080`)

## “소품 대체” 매핑

| 오프라인 소품        | 모바일/서버 대체                  |
| -------------- | -------------------------- |
| 캐릭터 토큰/시트      | 개인 화면 “역할 카드”(텍스트는 요약/링크)  |
| 그리모어(스토리텔러 보드) | 관리자 “그리모어 UI”(좌석/토큰/상태/메모) |
| 밤에 깨우기/선택      | 개인 화면 알림 + 선택 UI(서버로 제출)   |
| 투표 손들기         | 모바일 투표 버튼 + 타이머 + 집계       |
| 사망 표시 토큰       | 상태 토글(생존/사망) + 공지          |
| 타이머/벨          | 서버 타이머 브로드캐스트              |

## 기능 범위(MVP)

* [ ] 룸 생성/참가(QR 코드/6자리 코드)
* [ ] 플레이어 닉네임/좌석 확정
* [ ] 배역 배정(스크립트: Trouble Brewing 1종)
* [ ] 페이즈 제어: Night/Day/Execution
* [ ] 밤 행동 요청(대상 선택 1명 유형) + 제출
* [ ] 낮 투표 시작/종료 + 결과 표시
* [ ] 상태: 생존/사망, (옵션) 유령 투표 1회
* [ ] 관리자 로그(누가 언제 무엇을 선택했는지)

## 화면 정의(초안)

### 참가자

* `/join` : 방 코드 입력/QR 스캔
* `/me` : 내 역할/상태/공지
* `/night` : 밤 행동 UI(요청 시에만 활성)
* `/vote` : 투표 UI(투표 타임에만 활성)

### 스토리텔러

* `/admin` : 룸 대시보드
* `/admin/grimoire` : 좌석 원형 배치 + 토큰/상태
* `/admin/night` : 밤 순서 큐, 액션 트리거/확정
* `/admin/vote` : 투표 제어/집계/처형 확정

---

## 룰/운영 옵션(스토리텔러 설정값)

현장마다 운영 방식이 달라질 수 있어 옵션화합니다.

* 투표 방식: “지명자 1명에 대한 찬/반” vs “후보 다수 중 선택”
* 사망자 참여: 발언 가능/불가, 유령 투표 사용 여부(기본 룰은 설정에 따라 다를 수 있어 토글)
* 정보 노출: 역할 텍스트를 전부 노출 vs 요약 + 참조 링크
* 로그 공개: 스토리텔러만 vs 종료 후 리플레이 공개

---

### 네트워크 체크리스트

* 호스트 PC가 Wi-Fi에 연결되어 있고, 방화벽에서 포트 오픈
* 참가자 기기가 동일 네트워크에 연결
* 행사장 인터넷이 불안정하면 **외부망 없이도 되는 로컬 구성** 유지

## 보안/프라이버시

* 사내 PoC 기준 최소화
* 역할/비공개 정보는 “해당 플레이어 + 스토리텔러”만 접근
* 룸 코드는 충분히 랜덤/짧은 TTL 고려(행사 종료 시 폐기)
* 로그는 기본적으로 스토리텔러만 열람
